1. API GATEWAY
--------------------------------------------------------------------------------

[Câu] API versioning (v1/v2/test)
  Đáp án: Stages + Stage Variables
  Tip: Authorizer = auth; Resource Policy = access control — KHÔNG dùng versioning
  Tip: Mỗi stage có URL riêng; stage var trỏ Lambda alias

[Câu] prod/test/dev → Lambda khác nhau (PROD + DEV alias cùng lúc)
  Yêu cầu đề: 1 API Gateway, Lambda có alias PROD/DEV; cần 2 môi trường truy cập đồng thời, URL riêng
  Yêu cầu đề: Tạo stage PROD + stage DEV; mỗi stage có stage variable trỏ alias tương ứng
  Đáp án: Stage Variables + Lambda Aliases (D)
  Tip: ARN integration: ...function:MyFn:${stageVariables.lambdaAlias}
  Tip: "Environment variable" = Lambda config — KHÔNG phải tính năng API Gateway (bẫy C)
  Tip: Lambda Authorizer = auth; Gateway Response = custom error body — không route alias
  Lỗi hay nhầm: C — nhầm environment variable (Lambda) với stage variable (API GW)

[Câu] Method-level execution logging
  Đáp án: IAM role (trust apigateway.amazonaws.com) + STS enabled ở region
  Tip: Không nhầm execution logging vs access logging

[Câu] Custom headers + DynamoDB auth
  Đáp án: Lambda Authorizer (REQUEST type)
  Tip: Model = validate JSON; Cognito ≠ lookup DynamoDB custom

[Câu] HTTP error codes
  Đáp án: 503 = no target; 502 = bad response; 504 = timeout
  Tip: "Không biết gửi đi đâu" → 503

[Câu] Access control API Gateway — cơ chế KHÔNG hỗ trợ (NOT supported)
  Yêu cầu đề: Lambda behind API GW; đề hỏi cơ chế kiểm soát truy cập nào API GW không hỗ trợ
  Đáp án: STS — không phải authorizer/access mechanism của API Gateway
  Tip: Hỗ trợ: IAM SigV4, Cognito User Pools, Lambda Authorizer, API Keys, Resource policy
  Lỗi hay nhầm: Chọn IAM SigV4 — đây LÀ cơ chế hỗ trợ; STS chỉ service cấp temporary creds (dùng bởi IAM/role)

[Câu] Expose Lambda qua API GW — latency & cost thấp nhất, không cần API keys/usage plan/throttling
  Yêu cầu đề: Lambda cần public HTTP endpoint; ưu tiên rẻ + nhanh
  Yêu cầu đề: Không cần API keys, usage plans, per-method throttling
  Đáp án: HTTP API
  Tip: "lowest latency + cost" + "không cần tính năng nâng cao REST" → HTTP API
  Tip: HTTP API ~70% rẻ hơn REST API; latency thấp hơn; đủ cho Lambda proxy đơn giản
  Lỗi hay nhầm: Private REST API — "private" = chỉ truy cập trong VPC qua endpoint, không phải giải pháp rẻ/nhanh

[Câu] REST API 429 burst traffic — bảo vệ Lambda, serve tối đa request có thể
  Yêu cầu đề: REST API gặp 429 Too Many Requests khi burst; cần bảo vệ backend Lambda
  Yêu cầu đề: Vẫn phục vụ được nhiều request nhất có thể trong giới hạn an toàn
  Yêu cầu đề: Đề hỏi tính năng API Gateway — không phải cấu hình Lambda
  Đáp án: Usage plan với burst limit + rate limit
  Tip: 429 = API GW throttle; usage plan kiểm soát traffic TRƯỚC khi tới Lambda (token bucket)
  Tip: Rate limit = RPS ổn định; burst limit = chịu spike ngắn — tune để max throughput an toàn
  Lỗi hay nhầm: Caching — chỉ giảm load request trùng cache key, không bảo vệ burst đa dạng
  Lỗi hay nhầm: Provisioned concurrency — tính năng Lambda, không throttle request ở API GW
  Tip: Usage plan ≠ chỉ monetization — là cơ chế gắn API key + quota + rate/burst throttle; exam dùng để bảo vệ Lambda
  Tip: Không cần thu phí thật — chỉ cần API key + usage plan để kiểm soát RPS/burst theo client/tier

[Câu] Validate request trước Lambda — query string + headers bắt buộc, không sửa code Lambda
  Yêu cầu đề: Kiểm tra request TRƯỚC khi tới backend Lambda
  Yêu cầu đề: Required query string parameters và headers phải có mặt
  Yêu cầu đề: Không thay đổi code Lambda backend
  Đáp án: Request Validator (REST API)
  Tip: Request Validator = validate structure (body/query/headers có đủ không); Authorizer = auth ai được gọi
  Tip: Model gắn với Request Validator để validate JSON body schema; query/header chỉ cần Request Validator
  Lỗi hay nhầm: Lambda Authorizer — dùng cho auth/custom token lookup, không phải check param/header có mặt

[Câu] Troubleshoot HTTP 400 API Gateway — xác định nguyên nhân
  Yêu cầu đề: Client nhận HTTP 400 khi gọi endpoint; developer cần biết nguyên nhân
  Đáp án: Bật execution logging + access logging trên stage → CloudWatch Logs (tạo log group, gán ARN cho stage)
  Tip: 400 = Bad Request — thường validation (thiếu query/header/body sai schema) trước hoặc tại integration
  Tip: Execution log = chi tiết xử lý request/response integration; Access log = mỗi request (status, path, IP, latency)
  Tip: CloudTrail/CloudTrail Insights = API management calls (CreateApi, UpdateStage) — không log HTTP request client tới endpoint
  Lỗi hay nhầm: CloudTrail Insights để debug 400 client errors
  Lỗi hay nhầm: Firehose làm destination log API GW — chuẩn là CloudWatch Logs trực tiếp
  Lỗi hay nhầm: Chỉ X-Ray — tracing latency/service map; 400 validation thường thấy rõ hơn trong execution/access logs

[Câu] Q46 — Cognito Department + Lambda authorizer vẫn thấy Engineering
  Yêu cầu đề: API GW + Cognito federated IdP; attribute Department map sang Lambda authorizer; chỉ Sales được vào
  Yêu cầu đề: Đổi Department Engineering → Sales trên IdP; login lại vẫn deny; log/token vẫn Department=Engineering
  Đáp án: Authorization caching bật trên custom Lambda authorizer (API Gateway)
  Tip: API GW cache kết quả authorizer theo token/identity key (TTL 0–3600s); cùng token cũ → không gọi lại Lambda, vẫn deny cũ
  Tip: Token JWT giữ claim cũ đến khi hết hạn / re-auth hoàn toàn; cache authorizer làm “kẹt” quyết định cũ
  Lỗi hay nhầm: Authorization caching trên Cognito user pool — Cognito không có feature này
  Lỗi hay nhầm: IAM role Department tag — tag IAM không liên quan claim token/attribute IdP

2. LAMBDA
--------------------------------------------------------------------------------

[Câu] Node.js + dependencies (pg/RDS)
  Đáp án: Zip chung code + node_modules trong 1 file
  Tip: Lambda KHÔNG tự npm install; không upload 2 zip riêng

[Câu] Cross-account → DynamoDB (Account B)
  Yêu cầu đề: Lambda Account A cần đọc/ghi DynamoDB Account B
  Yêu cầu đề: Đề hỏi cấu hình IAM thế nào; DynamoDB không có resource policy
  Yêu cầu đề: Bẫy — sửa trust policy execution role A, hoặc policy trực tiếp lên table
  Đáp án: AssumeRole; trust policy trên role Account B
  Tip: Trust policy thuộc role ĐÍCH, không phải execution role A

[Câu] DLQ khi nào nhận message?
  Đáp án: Async invocation + fail HẾT mọi retry
  Tip: Sync (API GW) không dùng DLQ async; fail 1 lần rồi OK → không vào DLQ

[Câu] Async Lambda fail ngẫu nhiên — capture event fail (SQS DLQ)
  Yêu cầu đề: Lambda invoke async; một số event fail random; cần investigate + lưu event không xử lý được
  Yêu cầu đề: Cấu hình DLQ + execution role có quyền gửi message vào queue
  Đáp án: SQS standard dead-letter queue + update execution role (B)
  Tip: Lambda DLQ chỉ hỗ trợ SQS standard hoặc SNS — KHÔNG hỗ trợ FIFO queue
  Tip: Kinesis không phải DLQ target của Lambda; EventBridge→DynamoDB không phải pattern chuẩn capture fail
  Lỗi hay nhầm: C — SQS FIFO DLQ (Lambda không hỗ trợ FIFO làm DLQ)

[Câu] Async Lambda fail — cần event gốc + error details + function response
  Yêu cầu đề: Lambda xử lý async event từ S3; invocation fail; cần capture event gốc, chi tiết lỗi và function response để debug
  Yêu cầu đề: Đề hỏi cấu hình gì — không chỉ lưu event fail
  Đáp án: Lambda Destinations — On Failure destination trỏ SQS hoặc SNS
  Tip: DLQ chỉ gửi original event payload — KHÔNG có error details / function response
  Tip: Destinations On Failure gửi requestPayload + responseContext (error) + responsePayload
  Lỗi hay nhầm: Chọn DLQ khi đề nói rõ cần error details + function response

[Câu] Bundle dependencies
  Đáp án: 1 deployment package hoặc Lambda Layers

[Câu] Lambda Java — data >1MB encrypt/decrypt runtime
  Yêu cầu đề: Truyền >1MB data vào Lambda; encrypt/decrypt lúc runtime
  Đáp án: Envelope encryption + reference data as file trong code (deployment package / đọc file)
  Tip: Env var Lambda max ~4 KB tổng — không chứa 1MB; KMS Encrypt API max ~4 KB — không encrypt trực tiếp 1MB
  Lỗi hay nhầm: KMS/store env var; KMS direct encrypt file; envelope + env var

[Câu] Mobile game API — duplicate request / idempotency (retry throttling)
  Yêu cầu đề: Lambda REST API; mỗi request có unique ID; traffic spike ngẫu nhiên; client retry khi throttle
  Yêu cầu đề: Xử lý duplicate không mất dữ liệu / không inconsistent
  Đáp án: Lưu unique ID trong DynamoDB; Lambda check bảng trước khi xử lý
  Tip: Idempotency = check-or-store ID rồi mới process; DynamoDB scale serverless khớp Lambda
  Lỗi hay nhầm: Trả client error khi duplicate (client retry thêm); RDS (connection/scale); Memcached (volatile, mất key)

[Câu] Lambda SnapStart — giảm cold start (Java / Python / .NET)
  Yêu cầu đề: Lambda Java (hoặc Python 3.12+ / .NET 8) cold start chậm do JVM/runtime init; cần giảm latency khởi động mà không giữ instance luôn ấm
  Yêu cầu đề: Đề hỏi bật/cơ chế nào để tăng tốc cold start
  Đáp án: Bật SnapStart trên function → publish version; Lambda snapshot môi trường đã init rồi restore khi cold start
  Tip: SnapStart = snapshot sau init, restore nhanh; KHÔNG thay Provisioned Concurrency (giữ instance ấm 24/7)
  Tip: Runtime hỗ trợ: Java 11/17, Python 3.12+, .NET 8 — không phải mọi runtime
  Lỗi hay nhầm: Chọn Provisioned Concurrency khi đề chỉ cần giảm cold start chi phí thấp; tăng memory/timeout không fix JVM init

[Câu] EventBridge schedule → Lambda không chạy (rule enabled)
  Yêu cầu đề: EventBridge rule cron 9 AM UTC trigger Lambda; rule enabled nhưng Lambda không invoke
  Yêu cầu đề: Đề hỏi kiểm tra gì TRƯỚC TIÊN
  Đáp án: Lambda resource-based policy cho phép events.amazonaws.com invoke function
  Tip: Service invoke Lambda = resource policy trên function (ai được gọi); execution role = Lambda gọi service khác
  Tip: Tạo trigger qua Console thường auto thêm policy; tạo rule thủ công/IaC dễ thiếu
  Lỗi hay nhầm: Execution role events:PutEvents — Lambda không cần PutEvents để bị EventBridge invoke

[Câu] Traffic shifting Lambda — in-house deploy + alias (chọn 1)
  Yêu cầu đề: Lambda có alias; deploy bằng in-house solution (không CodeDeploy)
  Yêu cầu đề: Version mới chỉ nhận fixed % traffic production ban đầu
  Đáp án: Weighted alias — routing trên alias (AdditionalVersionWeights)
  Tip: Publish version mới → update alias: version chính + % weight cho version mới
  Tip: Canary/Linear deployment type = AWS CodeDeploy — không khớp in-house deploy
  Lỗi hay nhầm: Chọn CodeDeploy canary/linear khi đề nói in-house deployment solution
  Lỗi hay nhầm: Environment variables — không route traffic giữa versions

[Câu] Mobile app + API GW — test Lambda mới với subset user (LEAST effort)
  Yêu cầu đề: REST API → Lambda; test feature mới với một phần user, không ảnh hưởng user còn lại
  Yêu cầu đề: LEAST operational effort — giữ nguyên API Gateway, chỉ shift traffic ở Lambda
  Đáp án: Weighted alias — publish version mới + AdditionalVersionWeights trên alias (D)
  Tip: API GW integration trỏ alias ARN → chỉ update weight alias, không tạo REST API mới
  Tip: API GW stage canary = shift giữa API deployments, không phải cách ít effort nhất cho Lambda version
  Lỗi hay nhầm: C — tạo REST API mới + canary stage (overkill, effort cao)
  Lỗi hay nhầm: A — CodeDeploy canary cần setup deployment group (effort > weighted alias)

[Câu] External API key — Lambda config + full control KMS (CMK)
  Yêu cầu đề: Serverless app; lưu external API key như một phần Lambda configuration
  Yêu cầu đề: Full control KMS key encrypt API key; chỉ authorized entities thấy được
  Đáp án: Lambda environment variables + KMS customer managed key (CMK) encrypt env vars at rest
  Tip: "part of Lambda configuration" → Environment Variables; "full control KMS" → CMK không phải AWS managed/default
  Tip: Lambda encrypt env vars at rest bằng CMK; IAM + KMS key policy kiểm soát ai decrypt
  Lỗi hay nhầm: Parameter Store string + default AWS KMS key (A) — không phải Lambda config; default key = không full control
  Lỗi hay nhầm: Code repo / DynamoDB + AWS managed key (B/C) — secret không thuộc Lambda config; managed key ≠ full control

3. S3
--------------------------------------------------------------------------------

[Câu] Private bucket, truy cập tạm thời
  Đáp án: Pre-signed URL
  Tip: Bucket/IAM policy = quyền lâu dài

[Câu] External AWS account upload S3 tạm 2 giờ
  Yêu cầu đề: Account ngoài cần upload object vào bucket của bạn; access hết hạn sau 2h
  Yêu cầu đề: Temporary access — không chia long-term access keys
  Đáp án (2 đúng): (1) STS AssumeRole cross-account role + temp creds · (2) Pre-signed URL upload expiry 2h
  Tip: Pre-signed PUT URL — đơn giản cho upload có thời hạn; không cần external account assume role
  Tip: Cross-account role trust external account principal → AssumeRole → temp creds (session ≤ 12h)
  Lỗi hay nhầm: IAM user + access keys (long-term, anti-pattern); add external IAM user vào group account mình (không làm được)

  Đáp án: Bucket policy Deny + aws:SecureTransport: false
  Tip: KMS policy không enforce HTTPS cho GetObject

[Câu] Static website 403
  Đáp án: Bucket policy public s3:GetObject
  Tip: 403 = permission; CORS = cross-origin browser

[Câu] S3 mới, CloudFront cũ
  Đáp án: CloudFront invalidation
  Tip: CI/CD: upload S3 → invalidate CF

[Câu] Overwrite + đọc ngay
  Đáp án: Luôn trả version mới nhất (strong consistency từ 2020)
  Tip: Từ "might/propagate" = behavior cũ (SAI)

[Câu] Event notification thiếu
  Đáp án: Concurrent PUT cùng key, non-versioned → có thể chỉ 1 event
  Tip: Versioning GIÚP, không phải nguyên nhân thiếu event

[Câu] Cross-account NOT feasible
  Đáp án: Khác partition (aws vs aws-cn)
  Tip: Cùng partition → bucket policy / AssumeRole OK

[Câu] Multi-account upload, bucket owner
  Đáp án: S3 Object Ownership (Bucket owner enforced)
  Tip: CORS/Access Analyzer không đổi ownership

[Câu] Subset CSV (1 tháng, 3 cột)
  Yêu cầu đề: CSV lớn trên S3; lấy 1 tháng + 3/10 cột; minimize compute & network cost
  Đáp án: S3 Select
  Tip: Filter/projection trên S3 — không tải cả file; Analytics/Inventory/Access Logs không query nội dung CSV
  Lỗi hay nhầm: S3 Analytics (storage class); Inventory (metadata); Access Logs (audit)

[Câu] S3 access control (chọn 2)
  Đáp án: Bucket policy + IAM; Pre-signed URL + ACLs
  Tip: Permissions boundaries ≠ S3 access

[Câu] Transfer Acceleration
  Đáp án: Tăng tốc upload/download từ xa qua edge
  Tip: Không replicate; khác CloudFront cache

[Câu] On-prem upload file nén ~2 GB lên S3 backup hàng ngày — nhanh nhất
  Yêu cầu đề: Data center on-premises; file ~2 GB/ngày; hỏi cách upload S3 nhanh nhất
  Yêu cầu đề: File lớn + đường truyền từ xa → cần multipart VÀ tối ưu path
  Đáp án: Multipart upload + S3 Transfer Acceleration
  Tip: Multipart alone thiếu khi đề nhấn on-prem/xa; single PUT chậm; FTP→EC2→S3 thêm hop
  Lỗi hay nhầm: Chỉ multipart (không TA); single operation; relay qua EC2

[Câu] SSE-S3 vs SSE-KMS vs SSE-C
  Yêu cầu đề: Chọn loại mã hóa at rest; ai quản lý key
  Đáp án: SSE-S3 = S3 AES-256; SSE-KMS = KMS CMK; SSE-C = client gửi key mỗi request
  Tip: SSE-S3 đơn giản không KMS cost; SSE-KMS có audit CloudTrail
  Tip: SSE-C S3 không lưu key; bắt buộc HTTPS; mất key = mất data

[Câu] SSE-KMS — khóa KMS & quyền truy cập
  Yêu cầu đề: Access Denied object SSE-KMS; cross-account cần KMS key policy
  Đáp án: IAM cần S3 + kms:Decrypt/kms:GenerateDataKey; key policy allow principal
  Tip: aws/s3 default CMK vs customer managed CMK

[Câu] SSE-C — customer-provided keys
  Yêu cầu đề: Tự quản lý key ngoài AWS; header x-amz-server-side-encryption-customer-*
  Đáp án: Client gửi key mỗi PUT/GET; S3 không persist key
  Tip: Copy object SSE-C phải cung cấp key nguồn/đích

[Câu] Q91 — medical records S3: CMK + audit + rotate hàng năm
  Yêu cầu đề: Lưu hồ sơ y tế nhạy cảm trên S3; encrypt bằng customer-managed key; audit mọi key usage; rotate key hàng năm
  Đáp án: SSE-KMS với customer-managed CMK + bật automatic key rotation (annual) trên CMK
  Tip: CMK = customer managed key trong KMS; CloudTrail log kms:Decrypt/GenerateDataKey → auditable
  Tip: SSE-S3 = AWS-managed key, không CMK riêng, không audit chi tiết như KMS
  Lỗi hay nhầm: SSE-C — customer-provided key gửi mỗi request, không phải CMK; không có KMS audit/rotation tự động
  Lỗi hay nhầm: Client-side encryption local key — tự vận hành rotation/audit, không tích hợp KMS CloudTrail

[Câu] Default encryption & bắt buộc mã hóa
  Yêu cầu đề: Mọi upload phải SSE; không cho unencrypted object
  Đáp án: Default encryption bucket + bucket policy Deny nếu thiếu SSE header
  Tip: HTTPS (SecureTransport) ≠ SSE at rest — 2 policy khác nhau

[Câu] Storage classes — tổng quan
  Yêu cầu đề: Chọn class theo tần suất truy cập, retrieval time, chi phí
  Đáp án: Standard → IA → Glacier Instant/Flexible → Deep Archive
  Tip: Standard-IA/One Zone-IA min 30 ngày + retrieval fee
  Tip: Intelligent-Tiering auto tier, phí monitoring, không retrieval fee

[Câu] Chọn storage class theo use case
  Yêu cầu đề: Archive năm/chờ giờ/ms; pattern không rõ; chấp nhận 1 AZ
  Đáp án: Deep Archive rẻ nhất lâu dài; GIR ms retrieval; Intelligent-Tiering pattern không rõ
  Tip: One Zone-IA rẻ hơn Standard-IA nhưng chỉ 1 AZ durability

[Câu] Lifecycle — chuyển storage class
  Yêu cầu đề: Tự chuyển IA/Glacier sau N ngày; xóa sau retention
  Đáp án: S3 Lifecycle rules — Transition + Expiration
  Tip: Glacier restore tạo bản copy tạm; không transition ngược trực tiếp

[Câu] Q34 — S3 high request rates — phân tán prefix
  Yêu cầu đề: Nhiều object trong 1 bucket; cần optimize cho high request rates (PUT/GET)
  Đáp án: Object key names phân tán qua nhiều prefixes (vd user/12345/file, user/67890/file)
  Tip: S3 scale theo prefix trong key — mỗi prefix ~3500 PUT/s và ~5500 GET/s; root chung = bottleneck
  Tip: Prefix = phần path trong object key (trước tên file), KHÔNG phải object tag
  Lỗi hay nhầm: Object tag "prefix" — tag không ảnh hưởng request scaling
  Lỗi hay nhầm: Intelligent-Tiering — storage class/cost, không tăng request throughput
  Lỗi hay nhầm: Store ở root bucket — tất cả key cùng prefix rỗng → throttle

4. CLOUDFRONT
--------------------------------------------------------------------------------

[Câu] Route S3 theo country (cache miss)
  Đáp án: Lambda@Edge + origin request
  Tip: CloudFront Function chỉ viewer req/resp; origin request = cache miss

[Câu] Origin Groups failover — video streaming HA (chọn 2)
  Yêu cầu đề: CloudFront Origin Group primary + secondary; high availability / origin failover
  Yêu cầu đề: Chọn 2 câu ĐÚNG về hành vi failover — không nhầm cache hit hay “all primary”
  Đáp án (2): (1) Mỗi request mới vẫn thử primary trước — kể cả request trước đã failover sang secondary

  Tip: Primary Key = chỉ Partition Key (simple) HOẶC Partition Key + Sort Key (composite)
  Tip: Query BẮT BUỘC có Partition Key; Sort Key thêm điều kiện range/sort — KHÔNG thay thế Partition Key
  Lỗi hay nhầm: Sort Key thay Partition Key khi query; cả hai key đều optional (SAI — Partition Key bắt buộc)

[Câu] Tránh overwrite item đã tồn tại (cùng primary key)
  Yêu cầu đề: PutItem ghi đè item cùng PK → data discrepancy; cần ngăn overwrite
  Đáp án: Conditional writes — ConditionExpression attribute_not_exists(PK) hoặc attribute_not_exists(SK)
  Tip: PutItem mặc định overwrite toàn bộ item; condition = chỉ ghi khi thỏa điều kiện
  Lỗi hay nhầm: Atomic Counters (tăng số atomically, không chặn overwrite); BatchWrite (ghi hàng loạt, vẫn overwrite)

[Câu] Lambda upsert DynamoDB — minimum IAM permissions
  Yêu cầu đề: Lambda đọc item rồi update attribute hoặc tạo mới nếu chưa có (upsert); MINIMUM permissions

  Lỗi hay nhầm: Thêm PutItem (thừa); AddItem (không phải action IAM); GetRecords (Streams, không phải GetItem)

[Câu] Backup DynamoDB → S3 download local — NOT feasible (chọn 1)

  Đáp án: DynamoDB on-demand backup — backup lưu trong DynamoDB managed storage, restore vào table; KHÔNG ghi thẳng ra S3 để download
  Tip: Khả thi: EMR+Hive export, Data Pipeline export, Glue copy DynamoDB→S3; Export to S3 (tính năng riêng) ≠ on-demand backup
  Lỗi hay nhầm: Data Pipeline (thực ra khả thi export DDB→S3)

[Câu] BatchGetItem & BatchWriteItem

[Câu] Query vs Scan

  Đáp án: Query bắt buộc PK; Scan đọc hết — đắt RCU

[Câu] Scan table 100 GB — hành vi và chi phí RCU
  Yêu cầu đề: Developer Scan table 100 GB; đề hỏi behavior + cost concern
  Đáp án: Scan đọc MỌI item trong table bất kể FilterExpression — tốn RCU theo toàn bộ data scanned (~100 GB)
  Tip: FilterExpression chỉ lọc SAU khi đọc — không giảm RCU
  Tip: Scan KHÔNG tự dùng LSI optimize — LSI chỉ khác SK cùng PK; muốn dùng LSI phải Scan/Query trực tiếp index đó
  Lỗi hay nhầm: Scan dùng LSI tự optimize (C); Scan chỉ đọc item match filter (A); Scan rẻ hơn Query (D)

[Câu] projection / filter / condition expression — chat practice

  Đáp án: ProjectionExpression=attributes; FilterExpression=lọc items; ConditionExpression=điều kiện ghi

[Câu] DAX — DynamoDB Accelerator

  Đáp án: DAX cluster in-memory; read-through + write-through

[Câu] GSI vs LSI
  Yêu cầu đề: Query attribute không phải PK
  Đáp án: GSI PK/SK mới capacity riêng; LSI cùng PK khác SK lúc create table
  Tip: LSI max 5; strong read trên LSI

[Câu] GSI inverted index (PK/SK đảo) — query behavior
  Yêu cầu đề: GSI đảo PK/SK so với base table (table PK=userId SK=orderId; GSI PK=orderId SK=userId); đề hỏi khi query GSI expect gì
  Đáp án: GSI read eventually consistent + throughput provisioned riêng (RCU/WCU riêng)
  Tip: GSI KHÔNG hỗ trợ strongly consistent read — chỉ eventual; LSI mới có strong read
  Tip: GSI capacity tách khỏi base table — query GSI tốn RCU của GSI, không dùng RCU table
  Lỗi hay nhầm: Chọn GSI strongly consistent; hoặc GSI dùng chung throughput base table

[Câu] DynamoDB Streams + Lambda

[Câu] DynamoDB Streams — stream view type (old + new image)

  Đáp án: NEW_AND_OLD_IMAGES

[Câu] TTL — tự xóa item hết hạn
  Yêu cầu đề: Session/log auto-delete; attribute Number epoch; hoặc expiresAt đã qua 2 giờ — item còn trong table không?
  Đáp án: Enable TTL attribute; xóa nền không tốn WCU; item CÓ THỂ vẫn tồn tại — TTL deletion async, có thể mất tới 48 giờ
  Tip: Xóa không instant khi timestamp qua — không dùng TTL cho compliance xóa ngay lập tức
  Tip: Item hết hạn vẫn đọc được GetItem/Query/Scan cho đến khi DynamoDB xóa thật — không bị "ẩn"
  Lỗi hay nhầm: Xóa ngay khi expiresAt qua; item ẩn khỏi GetItem nhưng Scan vẫn thấy; chuyển sang S3 Glacier

[Câu] TransactWrite / TransactGet — ACID
  Yêu cầu đề: All-or-nothing nhiều item
  Đáp án: TransactWriteItems/TransactGetItems max 25, 4MB; write 2× WCU
  Tip: BatchWrite có thể partial success

[Câu] DynamoDB API — bảng tra cứu
  Đáp án: Put/Get/Update/Delete; BatchGet 100 BatchWrite 25; Query/Scan; Transact; PartiQL
  Tip: Pagination LastEvaluatedKey

[Câu] RCU / WCU — công thức cơ bản
  Yêu cầu đề: Cho kích thước item KB + read/write → hỏi cần bao nhiêu RCU/WCU
  Đáp án: WCU = ceil(KB/1); Strong RCU = ceil(KB/4); Eventual RCU = Strong ÷ 2
  Tip: 1 WCU = 1 KB write/s; 1 RCU strong = 4 KB read/s; 1 RCU eventual = 8 KB read/s
  Tip: Luôn ceil; transactional write = 2 × WCU

[Câu] Tính WCU — item lớn hơn 1 KB
  Yêu cầu đề: Item 2.5 KB, 6 KB ghi 1 lần/s → mấy WCU? BatchWrite cộng từng item
  Đáp án: WCU = ceil(kích thước KB); TransactWriteItems = 2 × WCU
  Tip: 2.5 KB → 3 WCU (không làm tròn xuống 2)

[Câu] Tính RCU — Strong vs Eventually consistent
  Yêu cầu đề: Cùng item so sánh RCU strong vs eventual; Scan/Query tính theo data đọc
  Đáp án: Strong RCU = ceil(KB/4); Eventual = Strong ÷ 2
  Tip: 10 KB strong = 3 RCU; eventual = 1.5 RCU
  Tip: Scan đọc cả table → RCU theo size scanned, không theo số row trả về

[Câu] Provisioned capacity — throttling
  Yêu cầu đề: ProvisionedThroughputExceededException; traffic spike vượt RCU/WCU
  Đáp án: Tăng RCU/WCU, Auto Scaling, hoặc chuyển On-Demand
  Tip: On-Demand tự scale, trả theo request; Provisioned rẻ hơn nếu traffic ổn định
  Tip: Chỉ áp dụng khi throttle do TỔNG throughput — KHÔNG fix hot partition (skew key)

[Câu] Insert DynamoDB provisioned — ProvisionedThroughputExceeded (chọn 2)
  Yêu cầu đề: App ghi (insert) vào DynamoDB provisioned; log báo ProvisionedThroughputExceededException
  Yêu cầu đề: Có thể nhắc EC2 burstable nano — đó là red herring; bottleneck là DynamoDB throughput
  Đáp án (2): Exponential backoff (giảm tần suất retry); Chuyển On-Demand capacity mode
  Tip: Insert/PutItem/UpdateItem = WCU — tăng RCU không giúp write throttle
  Tip: Giảm retry delay / gửi request dày hơn = làm throttle tệ hơn
  Tip: Đổi EC2 instance lớn hơn không tăng DynamoDB capacity
  Lỗi hay nhầm: Chọn tăng RCU khi đề nói insert/write

[Câu] Hot partition — 10 sensor chiếm 90% writes (IoT sensorId PK)
  Yêu cầu đề: Table PK = sensorId; vài key nhận phần lớn write → hot partition; đề hỏi cách phân tán write load
  Đáp án: Write sharding — thêm random suffix (1–N) vào partition key khi ghi; đọc scatter-gather qua các suffix
  Tip: Mỗi partition ~1000 WCU/s — hot key = 1 partition quá tải dù table còn capacity
  Tip: On-Demand / Auto Scaling scale tổng throughput — KHÔNG fix skew partition key
  Tip: GSI = access pattern query — không giải quyết hot write trên base table PK
  Lỗi hay nhầm: Chuyển On-Demand hoặc Auto Scaling khi đề nói hot partition / skew key

[Câu] GSI — capacity riêng
  Yêu cầu đề: Query nhiều trên GSI; write vào table có GSI
  Đáp án: GSI có RCU/WCU riêng; PutItem tốn WCU table + WCU GSI (projection)
  Tip: Thiếu GSI capacity → throttle riêng, không kéo theo base table

[Câu] Tham số SQS — tổng hợp nhanh

        import json
        def handler(event, context):
            return {'statusCode': 200, 'body': json.dumps({'msg': 'Hello'})}

    Code:
      S3Bucket: my-deploy-bucket

Multi-AZ (sync, HA)     vs Read Replica (async, read scale)
EFS (file, mount)       vs S3 (object) vs EBS (block, 1 EC2)
CloudFront Function     vs Lambda@Edge (có origin request)
Stage Variables         vs Lambda Authorizer
Execution logging       vs Access logging (API GW)
503 (no backend)        vs 502 (bad response) vs 504 (timeout)
Write Through           vs Lazy Loading + TTL
Gateway Endpoint        vs Interface Endpoint (S3/DDB = Gateway)
Trust policy            vs Permissions policy (cross-account)
ZipFile Node/Python (source text) vs ZipFile Java (base64 zip) vs S3Bucket+S3Key (package trên S3)
DeletionPolicy          vs UpdateReplacePolicy (xóa stack vs replace resource)
buildspec.yml           vs appspec.yml
CodeBuild               vs ASG (CB tự scale, không cần ASG)
Deployment Groups       vs Hooks (nhóm instance vs script lifecycle)
In-place deploy         vs Blue/Green (update vs swap traffic)
Fargate sidecar          vs EC2 Daemon Set (X-Ray Fargate = sidecar)
Task Execution Role     vs Task IAM Role (ECS)
SnapStart (snapshot restore, cold start nhanh) vs Provisioned Concurrency (instance luôn ấm, không cold start)
Lambda DLQ (chỉ original event) vs Lambda Destinations On Failure (event + error + response)
Lambda DLQ target: SQS standard hoặc SNS — KHÔNG FIFO, KHÔNG Kinesis
Partition Key (hash → physical partition) vs Sort Key (range → thứ tự trong partition)
Hot partition skew key (write sharding suffix) vs tăng capacity On-Demand/Auto Scaling
DynamoDB Stream view: KEYS_ONLY vs NEW_IMAGE vs OLD_IMAGE vs NEW_AND_OLD_IMAGES

22. LỖI HAY CHỌN NHẦM
--------------------------------------------------------------------------------

1. CloudFront Function + viewer request thay vì Lambda@Edge + origin request
2. S3 Standard-IA thay vì EFS Standard-IA (nhầm object vs file)
3. Multi-AZ async — Multi-AZ là SYNCHRONOUS
4. SSL pass-through thay vì SSL termination trên ALB
5. Metrics thay vì application logs (ECS Fargate)
6. Security Group thay vì IAM (authorization ECR)
7. CORS thay vì bucket policy (S3 static 403)
8. Trust policy execution role A thay vì role B (cross-account Lambda)
9. Parameter Store thay vì Secrets Manager (cross-account resource policy)
10. Lambda console trigger CodePipeline thay vì EventBridge
11. Connection pooling giảm DynamoDB latency — SAI
12. filter-expression thay vì projection-expression (chọn cột Scan)
13. Scan thay vì Query khi đã biết partition key
14. ASG / Auto Scaling cho CodeBuild — CB tự scale parallel
15. CodeDeploy Hooks thay vì Deployment Groups (nhóm EC2)
16. Cấu hình max revisions trên CodePipeline — phải config Agent trên EC2
17. CodePipeline promote khi CodeDeploy auto rollback — rollback = deployment mới
18. Detailed monitoring thay vì CodeBuild Agent local debug
19. Đổi Logical ID CFN nghĩ rename — Remove + Add, DB cũ có thể xóa
20. ECS Daemon Set cho X-Ray trên Fargate — Fargate dùng sidecar container
21. EB cross-account: share Saved Config qua IAM role / Import by name — phải download file + S3 account B
22. Origin Group: cache hit route primary / all origins primary — cache hit không gọi origin; chỉ 1 primary + 1 secondary
23. ECR pull local: docker login -u access key -p secret — phải get-login / get-login-password + docker pull URI
24. KCL max EC2 = 2× shard hoặc shard cũ — max worker song song = số shard hiện tại (1 shard = 1 lease)
25. KMS throttle: S3 bucket-level key — chỉ SSE-KMS trên S3; app bulk encrypt dùng Encryption SDK data key caching + backoff/retry
26. API Gateway access control: IAM SigV4 là hỗ trợ — STS không phải cơ chế authorizer riêng của API GW
27. Lambda async fail cần error details + response — DLQ chỉ gửi original event; dùng Lambda Destinations On Failure
28. DynamoDB hot partition / skew key — On-Demand hoặc Auto Scaling không phân tán write; dùng write sharding (random suffix + scatter-gather read)
29. DynamoDB Stream view type — KEYS_ONLY chỉ PK/SK; cần old + new image → NEW_AND_OLD_IMAGES
30. GSI query — eventually consistent only; throughput riêng; strong read chỉ base table / LSI
31. DynamoDB TTL — expiresAt qua rồi item vẫn có thể tồn tại tới 48h; không xóa instant; không ẩn khỏi GetItem
32. DynamoDB insert throttle — tăng RCU (SAI); insert cần WCU; đúng: exponential backoff + On-Demand (hoặc tăng WCU nếu đề ghi đúng loại)
33. CloudFormation giữ resource khi fail — termination-protection (SAI); đúng --disable-rollback trên create-stack/update-stack
34. Lambda traffic shifting in-house — CodeDeploy canary/linear (SAI); đúng weighted alias (AdditionalVersionWeights)
35. SAM gradual deploy — EventInvokeConfig OnSuccess/OnFailure (SAI); đúng DeploymentPreference + Canary10Percent30Minutes + Hooks
36. API GW troubleshoot 400 — CloudTrail Insights (SAI); đúng execution + access logging → CloudWatch Logs trên stage
37. Lambda Aurora connection string — IAM DB auth (SAI); đúng Parameter Store SecureString (single encrypted config)
38. CloudFront resize ảnh — chỉ max TTL (SAI); đúng Lambda@Edge on-demand + lưu variant S3; tránh Batch pre-gen all variants
39. API GW PROD/DEV Lambda alias — chọn "environment variable" (C); đúng stage variable + 2 stage (D)
40. Lambda subset user test — tạo REST API mới + API GW canary (C); đúng weighted alias (D, least effort)
41. Lambda async DLQ — chọn SQS FIFO DLQ (C); đúng SQS standard DLQ (B) — Lambda không hỗ trợ FIFO làm DLQ
